# IT运维自愈智能体 三期成熟生产版 完整落地手册

三期是一二期的**平滑升级版本**，核心定位是从「被动响应的单智能体」升级为「主动预防的多智能体运维团队」，不推翻原有架构，所有一二期的接口、应用、配置全部复用。它的核心目标是：故障主动预防、复杂故障协同解决、知识自动沉淀、自愈范围可控扩大，真正对标资深运维小组的能力。

> 说明：三期已不属于严格意义的MVP，而是生产环境的成熟迭代版，所有功能均遵循「先验证价值、再逐步放开、风险永远优先」的原则。

## 一、三期整体定位与能力边界

### 一二三期能力演进对比

| 维度   | 一期验证MVP     | 二期生产半自动化版    | 三期成熟生产版            |
|:---- |:----------- |:------------ |:------------------ |
| 核心模式 | 单智能体只读诊断    | 单智能体+低风险自愈   | 多智能体协同+主动预防        |
| 故障响应 | 人工喂告警，被动处理  | 告警自动流入，被动处理  | 主动巡检预警，提前干预        |
| 排障能力 | 通用基础故障      | 80%常见故障      | 覆盖系统/数据库/网络多领域复杂故障 |
| 自愈能力 | 无，仅给建议      | 一级低风险全自动     | 三级分级体系，二级人工确认执行    |
| 知识沉淀 | 无记忆，单次清零    | 人工上传知识库，被动复用 | 自动复盘生成案例，审核后自动入库   |
| 人力替代 | 替代30%重复排查工作 | 替代60%常规运维工作  | 替代80%常规运维+部分主动运维工作 |

### 三期核心验收指标

1. 复杂故障排障准确率≥92%，跨领域故障可自动分工协同
2. 主动预警覆盖率≥70%，30%潜在故障在触发告警前被发现并处理
3. 低风险故障自愈率≥50%，二级自愈操作零误操作
4. 知识库自动沉淀率≥80%，故障解决后24小时内完成知识入库
5. 单故障平均处理时长压缩至1.5分钟以内

## 二、三期必做4大核心模块（按优先级落地）

### 模块1：多智能体协同排障团队（核心架构升级）

#### 解决的痛点

二期单主Agent面对跨领域复杂故障（比如网络抖动+数据库慢查+服务异常的组合故障），容易出现判断偏差、专业度不足的问题；同时单Agent承载所有工具和逻辑，提示词臃肿，调试困难。

#### 整体架构：1个调度总管 + 3个专业子Agent

采用轻量化API调用方案实现，无需部署复杂的消息总线组件，完全基于现有Dify能力搭建，每个Agent职责单一、工具专属、提示词精简，调试和优化成本极低。

| 角色         | 核心职责                       | 专属工具                 | 专属知识库               |
|:---------- |:-------------------------- |:-------------------- |:------------------- |
| 调度总管Agent  | 接收故障、领域识别、任务分发、结果汇总、最终报告输出 | 调用3个子Agent的API工具     | 服务依赖关系库             |
| 系统专家Agent  | 服务器操作系统、资源、进程、磁盘类故障排查      | 服务器指标查询、进程分析、日志清理自愈  | Linux排障SOP、历史系统故障案例 |
| 数据库专家Agent | MySQL/Redis性能、连接、慢查询类故障排查  | 慢查询查询、连接数查询、索引优化建议   | 数据库故障案例库、SQL优化规范    |
| 网络专家Agent  | 网络延迟、丢包、端口不通、DNS类故障排查      | 端口连通性查询、延迟检测、DNS解析工具 | 网络故障排障SOP           |

#### 完整协同执行流程（真实故障演示）

输入故障：「核心业务接口间歇性超时，服务器CPU正常，数据库指标无明显异常」

1. 调度总管Agent接收故障，初步判断涉及多领域，拆解为3个子任务
2. 并行调用系统专家、数据库专家、网络专家三个子Agent同时排查
3. 各子Agent返回排查结果：
   - 系统专家：服务器资源正常，无异常进程
   - 数据库专家：慢查询数量正常，连接数充足
   - 网络专家：核心交换机端口存在丢包，丢包率3.2%
4. 调度总管汇总所有结果，判定根因为网络丢包，整合修复步骤，输出最终诊断报告
5. 自动推送钉钉告警，同步完整排查链路

#### Dify落地步骤

1. **创建3个专业子Agent应用**（2天）
   
   - 每个子Agent独立创建，仅配置对应领域的工具和知识库
   - 编写专属领域的系统提示词，聚焦单一专业方向，提升准确率
   - 每个子Agent开通API访问权限，获取API密钥和端点

2. **封装子Agent调用工具**（0.5天）
   在调度总管Agent的工具管理中，新建3个HTTP工具，分别对应3个子Agent的对话API，格式和二期预处理Agent调用主Agent的方式完全一致，复用现有逻辑。

3. **开发调度总管Agent**（1天）
   
   - 编写调度提示词，明确分工规则、结果汇总逻辑、异常兜底机制
   - 配置并行调用规则，提升排障速度
   - 增加反思校验：子Agent结论冲突时，自动发起二次复核

> 方案优势：完全复用Dify现有能力，无需额外部署组件；每个子Agent可独立优化，互不干扰；后续新增领域专家（如安全专家、中间件专家）成本极低。

### 模块2：主动巡检预警Agent（从被动救火到主动预防）

#### 解决的痛点

一二期都是故障触发告警后才介入处理，属于事后救火；三期通过定时主动巡检，识别增长性隐患（比如磁盘空间持续上涨、慢SQL数量逐渐增多），在故障爆发前提前预警处理，大幅降低业务影响。

#### 巡检覆盖范围（4大类，聚焦高价值隐患）

1. **系统资源类**：磁盘使用率增长趋势、内存使用率趋势、CPU负载趋势、inode使用率
2. **数据库类**：慢SQL数量趋势、连接数使用率、主从同步延迟、慢表空间增长
3. **应用服务类**：接口错误率趋势、响应时长趋势、线程池使用率
4. **安全基线类**：过期证书、弱口令账号、高危端口暴露

#### 核心能力：趋势预测+隐患分级

不是简单查看当前值，而是基于近7天数据做趋势预测，比如：

- 当前磁盘使用率75%，近7天每天增长2%，预测5天后达到90%告警阈值，提前发出预警
- 慢SQL数量从每天50条增长到每天200条，呈持续上升趋势，提前触发优化提醒

隐患分三级：

- 高危隐患：24小时内大概率触发故障，立即触发排障流程
- 中危隐患：3-7天内可能触发故障，纳入当日处理清单
- 低危隐患：长期增长趋势，纳入周度优化计划

#### 落地实现步骤

基于Dify工作流的定时触发器实现，无需额外开发定时任务。

1. **新建巡检工作流**（1天）
   
   - 在Dify中创建工作流应用，选择「定时触发器」作为启动节点
   - 配置执行周期：每天凌晨2点（业务低峰期）执行，避免影响业务
   - Cron表达式示例：`0 0 2 * * ?`，时区选择`Asia/Shanghai`

2. **开发巡检Agent**（2天）
   
   - 新建独立的巡检Agent，配置所有查询类工具
   - 编写巡检提示词，明确巡检项、趋势判断规则、分级标准
   - 输出结构化巡检报告，标注隐患等级和建议处理措施

3. **配置联动规则**（0.5天）
   
   - 高危隐患：自动调用调度总管Agent，触发深度排障
   - 中危隐患：推送钉钉告警，提醒当日处理
   - 低危隐患：纳入每日巡检汇总报告，统一推送

### 模块3：分级深度自愈体系（扩大自愈范围，风控升级）

#### 核心原则

永远不追求100%自动自愈，而是**风险与效率的平衡**：低风险全自动、中风险人工确认、高风险只给建议。二期已落地一级自愈，三期开放二级中风险操作，同时升级风控体系，确保零生产事故。

#### 三级自愈权限体系升级

| 等级  | 风险程度 | 操作内容                                      | 执行方式           | 三期是否开放       |
|:--- |:---- |:----------------------------------------- |:-------------- |:------------ |
| 一级  | 极低风险 | 清理过期日志、清理临时文件、清理Docker无用镜像、释放系统缓存         | 全自动执行，无需人工干预   | ✅ 二期已落地，三期延续 |
| 二级  | 中风险  | 非核心应用服务重启、Nginx配置重载、应用线程数扩容、非核心Docker容器重启 | 钉钉卡片人工确认，点击后执行 | ✅ 三期新增       |
| 三级  | 高风险  | 数据库重启、核心服务重启、杀进程、修改系统配置、删除业务数据            | 仅输出操作步骤，禁止自动执行 | ❌ 永久禁止自动执行   |

#### 风控升级4项强制机制

1. **人工确认机制**：所有二级操作，推送钉钉确认卡片，包含操作内容、影响范围、回滚方案，运维点击「确认执行」后才会操作
2. **自动回滚机制**：执行后自动校验服务状态，若30秒内服务未恢复正常，自动执行回滚操作（如回滚到上一个镜像版本），并推送告警
3. **熔断机制**：同一服务器1小时内自愈失败2次，自动禁用该服务器的所有自愈权限，推送人工介入，避免故障扩大
4. **双人复核机制**：涉及核心业务链路的二级操作，需要两位运维确认后才能执行

#### 落地步骤

1. **封装二级自愈接口**（2天）
   在现有FastAPI项目中扩展二级操作接口，每个接口都内置执行前校验、执行后状态检查、失败自动回滚逻辑
2. **钉钉审批卡片开发**（1天）
   对接钉钉互动卡片，实现「确认执行/拒绝/回滚」按钮交互，操作结果实时同步
3. **熔断逻辑实现**（0.5天）
   用轻量本地数据库记录执行结果，达到阈值自动触发熔断，同步更新白名单状态

### 模块4：故障自动复盘与知识自闭环

#### 解决的痛点

二期知识库需要人工整理案例、手动上传，效率低、容易遗漏，知识沉淀依赖人工自觉性；三期实现故障处理完后自动生成结构化复盘，人工审核后自动入库，形成「排障→复盘→入库→更准」的正向闭环，智能体越用越准。

#### 完整闭环流程

1. 故障处理完成，自愈/人工修复结束
2. 复盘Agent自动拉取全链路数据：告警信息、排查过程、根因结论、修复步骤、执行结果
3. 自动生成结构化故障复盘报告，包含：故障现象、根因、排查过程、修复方案、预防措施、经验总结
4. 推送运维人员审核，可修改补充内容
5. 审核通过后，自动调用Dify知识库API，将案例向量化存入历史故障案例库
6. 下次同类故障发生时，主排障Agent可自动召回该案例，直接复用解决方案

#### 技术实现

基于Dify知识库文档创建API实现自动入库，支持文本/文件两种方式上传，自动完成切片向量化。

1. **开发复盘Agent**（1天）
   独立复盘Agent，输入故障全链路数据，输出标准化Markdown格式的故障案例
2. **封装知识库上传接口**（1天）
   调用Dify官方`/datasets/{dataset_id}/document/create-by-file`接口，将生成的Markdown文件上传到指定知识库
   核心请求参数：
   - 知识库ID：对应历史故障案例库的ID
   - 文件：生成的Markdown复盘文件
   - 索引方式：高质量索引，自定义分段规则
3. **人工审核节点**（0.5天）
   增加人工审核步骤，审核通过后才触发入库，避免错误案例沉淀到知识库

## 五、三期必须遵守的风控与成本铁律

### 4条风控红线

1. **权限最小化**：所有自愈账号只授予最小必要权限，核心业务服务器永不加入自动执行白名单
2. **灰度全覆盖**：所有新增自愈操作，先在测试环境验证，再到非核心服务器灰度，最后逐步扩大范围
3. **一键熔断**：保留全局紧急开关，可一键关闭所有自动执行权限，降级为纯建议模式
4. **审计全追溯**：所有操作全链路留痕，包含操作人、时间、内容、结果、回滚记录，满足等保审计要求

### 3个成本控制技巧

1. **模型分级使用**：简单巡检、预处理用轻量模型，复杂排障、深度分析用强推理模型，整体token成本可降低40%
2. **按需调用子Agent**：调度总管先判断故障领域，只调用对应子Agent，不全部并行调用，减少无效消耗
3. **巡检错峰执行**：所有定时巡检放在业务低峰期，既不影响业务，也避开模型高峰时段的高并发延迟

## 六、核心验收测试用例

1. **多智能体协同验收**：输入跨领域复杂故障（网络丢包+服务超时），验证调度Agent能正确分发给网络专家，排除系统和数据库因素，最终输出准确根因
2. **主动巡检验收**：构造磁盘持续增长场景，验证巡检Agent能识别增长趋势，预测告警时间，提前发出中危预警
3. **二级自愈验收**：构造非核心服务异常场景，验证Agent推送确认卡片，人工确认后执行重启，服务异常时自动回滚，失败触发熔断
4. **知识闭环验收**：处理完一个新故障，验证自动生成复盘报告，审核通过后自动入库知识库，下次同类故障可自动召回参考

# 一、调度总管Agent 完整系统提示词（开箱即用，直接复制进Dify）

该Agent不做具体排障，核心职责是**故障领域识别、任务分发、多Agent结果汇总、冲突复核、输出最终报告**，是多智能体团队的“项目经理”。提示词严格约束边界，避免越权做专业领域的深度诊断。

```
# 角色
你是运维智能体团队的调度总管，负责统一接收故障告警与巡检隐患，识别故障所属领域，调度对应的专业子Agent协同排障，汇总所有子Agent的排查结果，输出最终的结构化诊断报告。你不直接执行具体排查操作，所有深度诊断均由对应领域的专业子Agent完成。

# 核心职责与子Agent分工规则
你可调度3名专业子Agent，分工严格对应，禁止跨领域调度：
1. 【系统专家Agent】
   - 负责领域：服务器操作系统故障、CPU/内存/磁盘资源异常、进程异常、文件系统问题、系统日志分析
   - 调度触发条件：出现服务器资源告警、进程异常、磁盘占满、系统负载过高等现象
2. 【数据库专家Agent】
   - 负责领域：MySQL/Redis性能问题、慢查询、连接数满、主从同步异常、SQL执行报错
   - 调度触发条件：出现慢查询超标、数据库CPU高、连接数占满、数据库报错等现象
3. 【网络专家Agent】
   - 负责领域：网络延迟、丢包、端口不通、DNS解析异常、超时类故障、链路抖动
   - 调度触发条件：出现接口间歇性超时、端口连通性异常、跨节点访问失败等现象

# 调度执行流程（严格遵循）
1. 第一步：故障领域研判
   接收故障描述后，先分析故障现象，判断涉及1个还是多个领域：
   - 单领域故障：仅调度对应1个子Agent，无需全部调用
   - 跨领域复合故障：并行调度所有涉及的子Agent，同时开展排查
   - 领域不明确：先并行调度3个子Agent做初步排查，再根据返回结果聚焦

2. 第二步：下发任务
   向子Agent传递完整的故障现象、告警信息、关联上下文，要求子Agent输出结构化的排查结论与证据。

3. 第三步：结果汇总与复核
   收到所有子Agent返回结果后，执行以下操作：
   - 梳理证据链：将各子Agent的结论按因果逻辑串联，区分根因与现象
   - 冲突校验：若多个子Agent结论矛盾，自动向对应子Agent发起二次复核，要求补充证据
   - 影响范围评估：结合故障类型，评估对业务的影响范围与严重等级

4. 第四步：输出最终报告
   整合所有信息，输出统一格式的最终诊断报告，附带修复优先级与执行建议。

# 强制反思校验（输出最终报告前必须执行）
1. 是否所有涉及的领域都已调度对应子Agent排查，有无遗漏？
2. 子Agent的结论是否有充分证据支撑，是否存在无依据的猜测？
3. 结论冲突的地方是否已完成二次复核，是否还有矛盾点？
4. 最终根因是否能解释所有故障现象，逻辑是否闭环？
5. 是否存在越权给出专业领域的具体操作命令？所有修复建议必须来自子Agent输出，不得自行编造。

# 输出规范（严格JSON格式，禁止额外自然语言）
{
    "故障概述": "一句话总结故障整体现象",
    "影响范围": "核心交易链路/单业务服务/边缘节点",
    "最终优先级": "P1/P2/P3",
    "根因结论": "明确的最终根因描述，置信度百分比",
    "排查分工明细": [
        {
            "执行Agent": "系统专家Agent",
            "排查结论": "该领域的排查结果",
            "关键证据": ["证据1", "证据2"]
        },
        {
            "执行Agent": "数据库专家Agent",
            "排查结论": "该领域的排查结果",
            "关键证据": ["证据1", "证据2"]
        }
    ],
    "修复步骤": ["步骤1", "步骤2", "步骤3"],
    "执行建议": "立即执行/工作日处理/低优先级优化",
    "风险提示": "修复操作的注意事项与回滚方案"
}

# 禁止行为
1. 禁止自行做深度专业诊断，所有排查结论必须来源于子Agent的返回结果
2. 禁止编造不存在的指标、日志、命令，所有信息必须有子Agent输出作为依据
3. 禁止遗漏任何子Agent的排查结果，所有参与排查的Agent必须体现在分工明细中
4. 禁止输出可直接执行的高危命令，仅输出修复步骤与执行建议
5. 禁止无关领域冗余调度，单领域故障不得调用全部子Agent，避免资源浪费
```

## 提示词优化补充说明

- 模型温度建议设置为 `0.1-0.2`，保证调度规则稳定，不随意发挥
- 建议开启Dify「结构化输出」功能，导入上述JSON Schema，强制输出格式合规
- 若业务有专属领域（如中间件、K8s），可在分工规则中新增对应子Agent，扩展成本极低

# 二、子Agent调用工具 详细配置清单（Dify分步操作）

## 前置准备工作

1. 确保3个专业子Agent（系统专家、数据库专家、网络专家）均已在Dify中创建完成，且单Agent排障功能调试正常
2. 为每个子Agent开通API访问权限，分别获取：
   - API端点（Endpoint），格式如 `https://你的Dify域名/v1/chat-messages`
   - API密钥（API Key）
3. 确保调度总管Agent与3个子Agent网络互通，Docker部署注意使用内网IP

---

## 通用配置模板（3个子Agent工具通用规则）

所有子Agent均封装为**自定义HTTP工具**，在调度总管Agent的【工具管理】中新建，配置规则统一：

- 请求方式：`POST`
- 请求头Headers：固定2个
  - `Authorization`：`Bearer 对应子Agent的API Key`
  - `Content-Type`：`application/json`
- 请求体Body：使用JSON格式，参数化传入故障信息
  
  ```json
  {
    "query": "{{fault_info}}",
    "response_mode": "blocking",
    "user": "scheduler-agent"
  }
  ```
  
  > 说明：使用`blocking`阻塞模式，等待子Agent返回完整结果后再继续处理，适合汇总场景；流式模式不适合调度汇总。

## 进阶优化配置

### 1. 并行调用优化

Dify Agent模式默认支持多工具并行调用，可在系统提示词中补充规则强化：

> 跨领域复合故障必须并行调用所有涉及的子Agent，同步执行，减少总排障时长。

### 2. 超时与重试配置

- 每个工具设置超时时间：建议30秒，避免子Agent长时间阻塞调度流程
- 增加1次失败重试：单次调用失败自动重试1次，仍失败则标记该领域排查异常，不影响其他领域结果汇总

### 3. 异常兜底规则

在提示词中补充兜底逻辑：

> 若某个子Agent调用失败，需在最终报告中标注「该领域排查异常，建议人工复核」，基于已成功返回的结果输出结论，不得因单个Agent失败导致整体排障中断。

# 三、联调验收测试用例

## 用例1：单领域故障（验证精准调度，不冗余调用）

- **输入**：`服务器192.168.1.20 根分区磁盘使用率95%，持续上涨`
- **预期行为**：仅调用「系统专家Agent」，不调用数据库和网络专家
- **预期输出**：根因为磁盘空间不足，修复步骤来自系统专家输出，分工明细仅包含系统专家

## 用例2：跨领域复合故障（验证并行调度+汇总）

- **输入**：`核心订单接口间歇性超时，数据库CPU偏高，服务器负载正常`
- **预期行为**：并行调用「数据库专家Agent」+「网络专家Agent」，系统专家不调用
- **预期输出**：汇总两个领域的排查结果，明确最终根因，分工明细包含两个Agent的结论

## 用例3：领域不明确故障（验证全量排查+聚焦）

- **输入**：`用户反馈网站访问慢，无具体告警信息`
- **预期行为**：并行调用3个子Agent做初步排查，再根据返回结果聚焦根因领域
- **预期输出**：最终定位到具体领域根因，分工明细包含所有Agent的初步排查结果
