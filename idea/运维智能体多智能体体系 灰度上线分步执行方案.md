# 运维智能体多智能体体系 灰度上线分步执行方案

本方案严格遵循**风险最小化、逐步验证、一键可回滚**原则，按照「先只读后执行、先边缘后核心、先影子后真实」的节奏推进，全程不影响现有运维体系，出现问题可秒级回退到人工处理模式。

## 一、灰度上线核心原则

1. **先软后硬**：先上线只读诊断能力（无业务影响），验证准确率达标后再逐步开放自动执行能力
2. **先边后核**：先覆盖非核心业务、测试环境、边缘节点，稳定后再向核心业务链路渗透
3. **小步快跑**：每个阶段设置明确的验收指标，不达标立即回退优化，达标再进入下一阶段
4. **全程可观测**：所有操作全链路留痕，灰度期间每日复盘准确率与异常案例
5. **核心红线**：核心数据库、核心交易服务器，永久只开放排障建议能力，不开放任何自动自愈权限

---

## 二、四阶段灰度上线全流程

### 阶段0：上线前准备（1天，前置必做）

#### 阶段目标

完成生产环境部署、配置校验、熔断开关测试，确保灰度开始前所有兜底机制生效。

#### 核心操作步骤

1. **生产环境部署**
   
   - 将FastAPI工具服务、Dify应用从测试环境发布到生产环境，使用独立的生产API Key
   - 配置独立的生产向量知识库，与测试环境数据隔离
   - 对接生产监控系统的备用Webhook通道，不影响原有告警链路

2. **灰度切流机制配置**
   
   - 在告警预处理Agent中新增**服务器IP白名单**配置，只有白名单内的服务器告警才进入智能体全链路处理
   - 不在白名单的告警，直接走原有告警通知流程，完全不经过智能体系统
   - 实现方式：在预处理Agent提示词中增加规则，或在转发脚本中增加IP过滤逻辑，无需改动核心代码

3. **熔断开关与回滚预案验证**
   
   - 测试全局一键降级开关：关闭所有自动自愈、关闭智能体调度，告警直接走原有人工链路
   - 测试模块级开关：可单独关闭自愈功能、单独关闭多智能体调度、单独关闭预处理聚合
   - 验证钉钉紧急通知通道，确保故障时可第一时间触达运维负责人

4. **基线数据采集**
   
   - 记录上线前一周的基线数据：平均排障时长、人工介入率、故障处理准确率
   - 用于后续灰度阶段的效果对比

#### 验收标准

- 所有接口生产环境连通性100%通过
- 三个熔断开关均测试有效，切换后10秒内生效
- 白名单过滤机制正常，非白名单告警不进入智能体链路
- 基线数据采集完成

---

### 阶段1：全量影子模式运行（3-7天，零业务风险）

#### 阶段目标

智能体全链路接收生产告警，但**只做分析、不做执行、不替代人工**，输出结果仅作为运维参考，验证排障准确率与稳定性。这是风险最低的验证阶段，建议停留时间最长。

#### 覆盖范围

- 告警范围：**全量服务器告警**都送入智能体系统（无需白名单限制）
- 能力范围：告警预处理 + 多智能体排障诊断，全部开启
- 执行限制：所有自愈操作强制 `dry_run=true`，只模拟不真实执行；不自动派单、不替代人工处理

#### 核心操作步骤

1. 监控系统开启双发：告警同时发给原有运维通道 + 智能体系统
2. 智能体完成全链路诊断后，将诊断报告推送到专属钉钉群，标注「影子模式-仅供参考」
3. 运维人员正常按原有流程处理故障，处理完成后与智能体结论做对比
4. 每日复盘：统计智能体根因判断准确率、建议有效性，记录错误案例，优化提示词与知识库

#### 验收通过标准（满足全部才可进入下一阶段）

1. 系统稳定性：智能体系统无宕机、无接口超时、告警无遗漏
2. 排障准确率：根因判断整体准确率 ≥ 90%，常见故障准确率 ≥ 95%
3. 误报率：错误诊断、误导性建议占比 ≤ 5%
4. 性能达标：单故障从告警接入到输出报告耗时 ≤ 2分钟

#### 建议停留时长

5-7个工作日，覆盖至少一次业务高峰期与夜间低峰期

#### 风险与注意事项

- 本阶段零业务风险，智能体完全旁路运行，不影响任何现有运维流程
- 重点关注告警风暴场景下的系统稳定性与token成本
- 禁止在本阶段开放任何真实执行权限

---

### 阶段2：小流量自动自愈灰度（7-10天，低风险验证）

#### 阶段目标

在非核心服务器上开放一级低风险自动自愈能力，验证自愈执行的准确性、安全性与风控机制。

#### 覆盖范围/切流比例

- 服务器范围：选取 **3-5台非核心边缘服务器** 加入白名单（如日志服务器、测试环境服务器、静态资源服务器）
- 自愈权限：仅开放**一级极低风险操作**（清理过期日志、清理临时文件、清理Docker无用镜像）
- 排障能力：白名单内服务器的全量告警，均进入多智能体排障流程

#### 核心操作步骤

1. 将首批试点服务器IP加入白名单，关闭这些服务器自愈操作的 `dry_run` 模式，开启真实执行
2. 执行规则：
   - 所有自愈操作执行前，自动推送钉钉预告通知
   - 执行后立即推送执行结果，包含删除文件数、释放空间、执行时长
   - 审计日志同步记录，每日人工核对执行记录
3. 每日核查：核对自愈操作记录，确认无误删、无越权、无业务影响
4. 逐步扩容：每2天新增2-3台非核心服务器，逐步扩大白名单范围

#### 验收通过标准

1. 自愈执行准确率100%，无误删、无越权操作，无业务影响
2. 风控机制全部生效：白名单外服务器无法触发自愈，路径越权被拦截
3. 审计日志完整可追溯，所有操作100%留痕
4. 对应故障的自动解决率 ≥ 80%

#### 建议停留时长

7-10个工作日，累计执行自愈操作不少于30次，无异常

#### 风险与注意事项

- 严格控制白名单范围，核心业务服务器绝不进入本阶段
- 保留一键关闭自愈功能的开关，出现异常立即降级为纯建议模式
- 所有删除类操作保留24小时备份，发现误删可快速恢复

---

### 阶段3：分级全量上线（持续迭代，生产正式运行）

#### 阶段目标

非核心业务全量覆盖，核心业务开放只读诊断，二级自愈开放人工确认模式，智能体正式成为运维团队的辅助生产力工具。

#### 覆盖范围/切流比例

1. **排障诊断能力**：100%全量服务器告警进入智能体预处理+排障流程，输出诊断报告作为运维参考
2. **一级自愈能力**：所有非核心服务器全部开放，覆盖约80%的服务器节点
3. **二级自愈能力**：选取部分非核心应用服务，开放**人工确认执行**模式（钉钉卡片点击确认后执行重启/重载）
4. **核心业务服务器**：仅开放排障诊断建议，永久不开放自动自愈权限

#### 核心操作步骤

1. 移除排障诊断的白名单限制，全量告警接入智能体处理
2. 扩大一级自愈白名单至所有非核心服务器
3. 上线二级自愈人工确认卡片，针对非核心服务的重启类操作，推送确认卡片，运维点击后执行
4. 建立周度优化机制：每周复盘错误案例，更新知识库与提示词，持续提升准确率
5. 输出运维规范：明确智能体的职责边界、人工复核场景、异常上报流程

#### 验收通过标准

1. 整体运维效率提升：常规故障平均处理时长下降 ≥ 70%
2. 人工介入率下降：60%以上的常规故障无需人工深度排查
3. 自愈覆盖度：50%以上的低风险故障可自动解决
4. 连续14天无生产安全事故、无误操作

#### 建议停留时长

长期运行，持续迭代优化，进入常态化运营阶段

## 三、分级回滚与紧急熔断方案

### 1. 三级回滚机制

| 回滚级别  | 触发场景                  | 回滚操作                                | 生效时间 |
|:----- |:--------------------- |:----------------------------------- |:---- |
| 模块级回滚 | 单个功能异常（如自愈误判、预处理聚合错误） | 单独关闭对应模块功能，其他能力正常运行                 | 10秒内 |
| 执行级回滚 | 自愈操作出现异常、有误操作风险       | 一键关闭所有自动执行权限，全局降级为纯建议模式             | 10秒内 |
| 全局回滚  | 智能体系统严重故障、影响告警正常接收    | 切断智能体告警入口，所有告警完全走原有人工处理链路，智能体系统整体下线 | 30秒内 |

### 2. 一键熔断开关实现方式

无需复杂开发，通过3个简单开关即可实现：

1. **配置文件开关**：在FastAPI服务的配置文件中设置 `self_healing_enabled=true/false`，控制自愈接口是否真实执行
2. **告警转发开关**：在告警转发脚本中设置 `agent_enabled=true/false`，控制是否将告警转发给智能体
3. **Dify应用开关**：直接停用对应Dify应用，API调用立即返回失败，告警自动走兜底流程

### 3. 自愈操作回滚预案

针对已执行的自愈操作，提前准备回滚手段：

- 日志/文件清理：删除前先移动到 `/backup/` 临时目录，保留24小时，发现误删可立即恢复
- 服务重启：保留上一个版本的启动脚本，执行后30秒未恢复自动回滚到上一状态
- 配置重载：执行前自动备份配置文件，异常时一键还原

## 四、灰度期核心监控与评估指标

灰度期间每日统计以下指标，作为是否进入下一阶段的判断依据：

### 1. 业务效果指标

| 指标      | 说明                   | 合格阈值  |
|:------- |:-------------------- |:----- |
| 根因判断准确率 | 智能体判断的根因与人工最终结论一致的比例 | ≥ 90% |
| 自动自愈成功率 | 自动执行的自愈操作，成功解决问题的比例  | ≥ 95% |
| 平均排障时长  | 从告警产生到输出完整诊断报告的时间    | ≤ 2分钟 |
| 人工介入率   | 需要人工深度排查的故障占比        | ≤ 40% |

### 2. 系统稳定性指标

| 指标     | 说明              | 合格阈值    |
|:------ |:--------------- |:------- |
| 系统可用性  | 智能体服务正常运行时间占比   | ≥ 99.9% |
| 告警漏处理率 | 进入系统但未生成报告的告警占比 | ≤ 0.1%  |
| 接口响应耗时 | 底层工具接口平均响应时间    | ≤ 500ms |
| 异常崩溃次数 | 每日系统异常崩溃次数      | 0次      |

### 3. 成本指标

| 指标        | 说明               | 合格阈值   |
|:--------- |:---------------- |:------ |
| 日均token消耗 | 大模型API每日总token量  | 符合预算预期 |
| 单故障平均成本   | 处理单条故障的平均token费用 | 可接受范围内 |
| 无效调用占比    | 冗余、错误的工具调用占比     | ≤ 10%  |

### 4. 安全风控指标

| 指标      | 说明              | 合格阈值 |
|:------- |:--------------- |:---- |
| 越权操作拦截率 | 尝试越权的操作被成功拦截的比例 | 100% |
| 审计日志完整率 | 所有操作都有审计记录的比例   | 100% |
| 误操作次数   | 对业务造成影响的误操作次数   | 0次   |

---

## 五、灰度期日常运营机制

1. **每日短会**：10分钟站会，复盘前一天的错误案例、异常告警，同步优化进度
2. **案例入库**：所有人工处理的新故障，24小时内整理成结构化案例，审核后入库知识库
3. **规则迭代**：每周优化一次提示词与工具描述，针对高频错误场景定向优化
4. **月度复盘**：对比基线数据，评估效率提升效果，制定下一阶段优化目标
