# 多智能体全链路联调测试手册（分4阶段，端到端全覆盖）

整体遵循**分层递进、从底到上**的测试原则：先验证单点工具与单Agent能力，再测试多Agent协同调度，最后跑通全链路端到端流程，出现问题可快速定位层级，避免全链路混测难以排查。
所有用例基于前文的模拟接口即可完成全量测试，无需接入真实生产环境。

---

## 第一阶段：底层工具接口连通性测试（前置必测）

所有上层Agent能力都依赖底层接口，本阶段100%通过后再进入后续测试，核心验证「接口通、参数对、风控生效」。

### 1. 正常功能验证

在服务器本地执行curl命令，逐一验证所有新增接口返回正常：

| 接口名称     | 测试命令示例                                                                                                     | 预期结果                                           |
|:-------- |:---------------------------------------------------------------------------------------------------------- |:---------------------------------------------- |
| 磁盘目录占用分析 | `curl "http://127.0.0.1:8001/api/ops/system/disk-directory-analyze?server_ip=192.168.1.20&path=/var/log/"` | 返回JSON格式，包含top_files、sub_dir_sorted字段，数据与模拟值一致 |
| 进程详情查询   | `curl "http://127.0.0.1:8001/api/ops/system/process-detail?server_ip=192.168.1.20&pid=1234"`               | 返回进程名、运行用户、资源占用等字段                             |
| 数据库连接数查询 | `curl "http://127.0.0.1:8001/api/ops/database/connection-status?instance=master"`                          | 返回连接总数、使用率、来源TOP5                              |
| 主从同步延迟查询 | `curl "http://127.0.0.1:8001/api/ops/database/replication-delay?instance=slave"`                           | 返回同步状态、延迟秒数字段                                  |
| 端口连通性检测  | `curl "http://127.0.0.1:8001/api/ops/network/port-check?target_ip=192.168.1.20&port=3306"`                 | 返回is_open、响应耗时字段                               |
| 网络延迟丢包检测 | `curl "http://127.0.0.1:8001/api/ops/network/ping-check?target_ip=192.168.1.20"`                           | 返回丢包率、平均延迟字段                                   |
| DNS解析测试  | `curl "http://127.0.0.1:8001/api/ops/network/dns-check?domain=mysql.service.local"`                        | 返回解析IP列表、解析耗时字段                                |

### 2. 安全风控验证

核心验证白名单机制生效，防止越权调用：

1. **IP白名单拦截**：传入不在白名单的IP，预期返回403错误与拦截提示
2. **路径白名单拦截**：传入不在白名单的目录（如`/root/`），预期返回403
3. **路径遍历攻击防护**：传入`path=/var/log/../etc/`，预期返回403非法路径提示
4. **端口范围校验**：传入端口0或65536，预期返回400参数错误

## 第二阶段：单Agent独立功能验收测试

分别对3个专业子Agent单独调试，验证「工具调用准确、推理逻辑正确、输出格式合规、边界不越权」，确保每个子Agent自身能力达标后，再接入调度总管。

### 一、系统专家Agent 测试用例

#### 用例1：磁盘占满故障排查

- **输入**：`服务器192.168.1.20根分区磁盘使用率95%，请排查原因`
- **预期工具调用**：先调用「服务器指标查询」，再调用「磁盘目录占用分析」
- **预期输出要点**：
  1. 定位日志目录占用最高，识别大日志文件
  2. 给出清理过期日志的修复步骤
  3. 标记「可自动自愈」，说明调用日志清理工具
- **验收标准**：工具调用顺序符合排查逻辑，结论有数据支撑，JSON格式完整。

#### 用例2：CPU高进程异常排查

- **输入**：`192.168.1.20 CPU使用率96%，持续10分钟`
- **预期工具调用**：先调用「服务器指标查询」获取TOP进程，再调用「进程详情查询」查看异常进程
- **预期输出要点**：
  1. 明确mysqld进程占用CPU最高
  2. 给出进程详情与业务归属
  3. 建议调度数据库专家进一步排查数据库层原因
- **验收标准**：能识别异常进程，且不越权做数据库深度诊断，主动建议调度对应领域专家。

#### 用例3：证据不足边界场景

- **输入**：`服务器服务异常退出，没有其他信息`
- **预期输出要点**：
  1. 明确说明「当前数据不足无法精准定位根因」
  2. 给出人工排查步骤（查看系统日志、检查服务配置）
  3. 不编造进程、日志信息
- **验收标准**：如实说明能力边界，不强行给出结论，符合反思闭环要求。

#### 用例4：跨领域越权拦截

- **输入**：`帮我优化一下数据库的慢SQL`
- **预期输出要点**：说明自身仅负责系统层问题，建议调度数据库专家Agent处理，不给出具体SQL优化方案
- **验收标准**：严格遵守领域边界，不越权输出非自身领域的专业结论。

### 二、数据库专家Agent 测试用例

#### 用例1：慢SQL性能故障排查

- **输入**：`MySQL主库CPU高，慢查询数量突增，请排查`
- **预期工具调用**：调用「MySQL慢查询查询」
- **预期输出要点**：
  1. 列出Top慢SQL，标注执行时长与扫描行数
  2. 给出添加索引的优化步骤
  3. 提示低峰期执行，标注灰度执行风险
- **验收标准**：优化建议合理，风险提示明确，有数据支撑。

#### 用例2：连接数耗尽故障排查

- **输入**：`数据库连接数占满，应用无法建立连接`
- **预期工具调用**：调用「数据库连接数查询」
- **预期输出要点**：
  1. 说明连接使用率、主要来源服务
  2. 给出临时扩容连接数、排查连接泄露的分步建议
  3. 标记紧急程度，建议立即处理
- **验收标准**：区分临时缓解与根治方案，逻辑清晰。

#### 用例3：主从同步延迟排查

- **输入**：`从库查询数据延迟，排查主从同步问题`
- **预期工具调用**：调用「主从同步延迟查询」
- **预期输出要点**：
  1. 返回同步状态、延迟时长
  2. 给出常见延迟原因与排查方向
  3. 延迟为0时明确说明同步正常，建议排查其他原因
- **验收标准**：状态判断准确，排查步骤符合运维规范。

### 三、网络专家Agent 测试用例

#### 用例1：端口不通故障排查

- **输入**：`应用无法连接192.168.1.20的3306端口，请排查`
- **预期工具调用**：调用「端口连通性检测」
- **预期输出要点**：
  1. 返回端口通断状态与响应耗时
  2. 给出排查方向：防火墙、安全组、服务是否启动
  3. 明确故障边界
- **验收标准**：结论基于检测结果，排查步骤分层清晰。

#### 用例2：网络丢包延迟排查

- **输入**：`跨机房接口间歇性超时，怀疑网络问题`
- **预期工具调用**：调用「网络延迟丢包检测」
- **预期输出要点**：
  1. 返回丢包率、平均延迟、抖动数据
  2. 判断链路质量等级
  3. 给出对应排查建议：运营商链路、交换机端口
- **验收标准**：质量分级准确，建议匹配故障现象。

#### 用例3：非网络问题边界验证

- **输入**：`网站访问慢，帮我查一下是不是网络的问题`
- **预期工具调用**：调用ping检测、端口检测，结果均正常
- **预期输出要点**：
  1. 明确说明「网络层检测正常，排除网络因素」
  2. 建议调度系统专家、数据库专家进一步排查业务层
- **验收标准**：排除自身领域问题后明确说明边界，不强行归因网络。

## 第三阶段：调度总管Agent 协同能力测试（三期核心验收）

本阶段验证多智能体架构的核心价值：精准调度、并行协同、结果汇总、异常兜底，全部通过才算多Agent架构生效。

### 用例1：单领域故障 → 精准调度（无冗余调用）

- **输入**：`服务器192.168.1.30 磁盘使用率92%`
- **预期调度行为**：仅调用「系统专家Agent」，不调用数据库、网络专家
- **预期输出要点**：
  1. 排查分工明细仅包含系统专家
  2. 最终根因、修复步骤与系统专家输出一致
  3. 无冗余调用，token消耗最低
- **验收标准**：调度精准，无无效调用，符合成本控制规则。

### 用例2：跨领域复合故障 → 并行调度+结果汇总

- **输入**：`核心订单接口间歇性超时，数据库CPU偏高，服务器整体负载正常`
- **预期调度行为**：并行调用「数据库专家Agent」+「网络专家Agent」，不调用系统专家
- **预期输出要点**：
  1. 排查分工明细包含两个Agent的独立结论与证据
  2. 最终整合出统一根因，区分主次因素
  3. 修复步骤按优先级排序
- **验收标准**：两个Agent并行执行，总耗时接近单Agent时长；汇总逻辑清晰，不简单堆砌结果。

### 用例3：领域不明确故障 → 全量排查+聚焦

- **输入**：`用户反馈网站访问很慢，没有具体告警信息`
- **预期调度行为**：并行调用3个子Agent做初步排查，再根据结果聚焦根因领域
- **预期输出要点**：
  1. 分工明细包含三个Agent的初步排查结果
  2. 最终定位到具体领域根因，排除其他领域
  3. 排查逻辑清晰，符合分层排查原则
- **验收标准**：领域不明时不遗漏方向，初步排查后能快速收敛范围。

### 用例4：子Agent结论冲突 → 自动二次复核

- **测试构造**：模拟两个Agent结论矛盾（如网络专家说链路正常，应用专家说网络超时）
- **预期调度行为**：调度总管识别冲突，自动向对应Agent发起二次复核，要求补充证据
- **预期输出要点**：
  1. 最终报告中标注「结论存在冲突，已二次复核」
  2. 补充复核后的最终结论与补充证据
  3. 不直接采信某一方结论，有验证过程
- **验收标准**：具备冲突识别与复核能力，符合反思闭环要求。

### 用例5：单Agent调用失败 → 异常兜底不中断

- **测试构造**：临时关闭网络专家Agent的API，模拟调用失败
- **预期调度行为**：重试1次仍失败后，标记该领域排查异常，基于其他Agent结果继续输出
- **预期输出要点**：
  1. 分工明细中标注该领域「排查异常，建议人工复核」
  2. 基于已成功的Agent结果，输出部分可用的结论
  3. 不因单个Agent失败导致整体排障中断
- **验收标准**：具备容错能力，局部故障不影响整体流程，符合生产可用性要求。

## 第四阶段：全链路端到端验收测试（真实生产流程模拟）

从告警输入到最终推送，完整走通「告警预处理Agent → 调度总管Agent → 专业子Agent → 自愈执行 → 钉钉推送」全流程，模拟真实生产场景。

### 用例1：P1级跨领域复杂故障全流程

- **完整输入**（模拟监控推送原始告警流）：
  
  ```
  【P2】192.168.1.20 CPU使用率96%
  【P2】订单接口响应超时率18%
  【P1】MySQL主库慢查询260条/分钟
  【P2】支付接口超时率12%
  【P3】日志服务磁盘使用率85%
  ```
- **预期全链路执行步骤**：
  1. 告警预处理Agent：去重聚合，生成2个故障组（P1核心数据库故障组、P3磁盘告警组）
  2. P1故障组自动触发调度总管Agent，调度数据库专家+系统专家排查
  3. 子Agent返回结果，调度总管汇总输出最终诊断报告
  4. 调用钉钉推送工具，P1级@负责人，推送完整报告
  5. P3级告警不触发深度排障，纳入待汇总清单
- **验收标准**：全流程无需人工干预，自动完成，最终钉钉收到格式正确的P1告警消息，@对应负责人。

### 用例2：P2级单领域可自愈故障全流程

- **完整输入**：
  
  ```
  【P2】192.168.1.21 磁盘使用率94%，持续上涨
  ```
- **预期全链路执行步骤**：
  1. 预处理Agent识别为单P2故障，自动触发深度排查
  2. 调度总管调度系统专家Agent
  3. 系统专家诊断为过期日志占用，标记可自动自愈
  4. 自动调用日志清理自愈工具（试运行模式/真实执行模式）
  5. 执行完成后，推送钉钉结果通知，包含释放空间、删除文件数
- **验收标准**：从告警到自愈执行完成全自动化，执行结果可审计，钉钉推送内容完整。

### 用例3：P3级低优先级告警全流程

- **完整输入**：
  
  ```
  【P3】测试环境日志服务磁盘使用率82%
  ```
- **预期全链路执行步骤**：
  1. 预处理Agent识别为P3低优先级、测试环境告警
  2. 不触发深度排障，不调用子Agent
  3. 纳入每日告警汇总清单，不单独推送
- **验收标准**：不产生无效的深度排障调用，成本控制规则生效。

## 第五阶段：异常容错场景测试

### 用例1：单个工具调用失败

- **测试方式**：临时关闭某个查询接口，模拟工具异常
- **预期结果**：Agent标记该工具调用失败，基于已有数据输出结论，明确说明数据缺失，不编造信息。

### 用例2：输入异常/无效告警

- **输入**：乱码、无意义文本、无关问题
- **预期结果**：预处理Agent识别为无效告警，过滤并说明原因，不触发后续排障流程。

### 用例3：自愈操作白名单拦截

- **测试方式**：传入不在白名单的服务器IP，调用自愈接口
- **预期结果**：拦截操作，返回权限不足提示，不执行删除动作，审计日志记录违规尝试。

## 最终验收通过标准

全部满足以下条件，即为三期多智能体联调通过，可进入灰度上线阶段：

1. 所有底层工具接口连通性100%正常，安全风控规则全部生效
2. 3个专业子Agent单领域排障准确率≥95%，输出格式合规率100%
3. 调度总管调度准确率≥90%，跨领域故障汇总逻辑清晰，具备冲突复核与异常兜底能力
4. 3条端到端全链路用例全部跑通，从告警输入到结果推送无人工干预
5. 异常场景下系统不崩溃、不误操作，符合生产可用性要求
6. 所有操作均有审计日志，可追溯全链路执行过程
