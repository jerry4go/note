### 告警降噪聚合 Agent

#### 最小实现方案（纯推理，无需额外工具）

1. 在 Dify 里新建一个独立的 Agent 应用，命名为「告警预处理 Agent」；
2. 编写专属系统提示词，只给它降噪、聚合、分级的规则，不给排障工具；
3. 监控系统的 Webhook 先推送到这个 Agent，而不是主排障 Agent；
4. 预处理完成后，自动调用主排障 Agent 的 API，传入聚合后的故障组。

### 进阶增强方案（准确率更高）

给它增加 1 个工具：**CMDB 服务依赖查询接口**，让它可以查询服务之间的上下游关系，聚合连锁告警时更精准，比如知道订单服务依赖数据库，就能判断订单超时大概率是数据库故障引发的。



## 一、告警预处理 Agent 完整系统提示词（开箱即用，直接复制进 Dify）

这个 Agent 的定位是**纯前置分诊，不做深度排障、不调用运维工具**，只负责告警清洗、聚合、分级，输出结构化故障组后再调度主排障 Agent。提示词严格约束边界，避免越权做深度推理。

具体提示词看  告警预处理 Agent 完整系统提示词.txt  文件
二、Dify 全链路串联配置分步指引
==================

实现完整数据流：**监控系统 → 告警预处理 Agent → 主排障 Agent → 推送运维人员**，全程自动化，无需人工中转。
第一步：创建告警预处理 Agent 应用

--------------------

1. 进入 Dify 后台 → 【创建应用】→ 选择【Agent】类型，命名为「告警预处理 Agent」；
2. 在【编排】→【系统提示词】中，粘贴上方完整提示词；
3. 【响应设置】中开启「结构化输出」，导入上述 JSON Schema，强制模型按格式输出；
4. 模型温度调至 `0.1-0.2`，降低随机性，保证聚合规则稳定。

第二步：配置「调用主排障 Agent」的 HTTP 工具
----------------------------

让预处理 Agent 完成聚合后，自动调用主排障 Agent 进行深度诊断，本质是把主排障 Agent 的 API 封装成自定义工具。

### 1. 获取主排障 Agent 的 API 信息

1. 打开你之前创建的「运维自愈智能体」（主排障 Agent）；

2. 左侧菜单进入【发布】→【API 访问】，点击【创建 API 密钥】，生成并复制 Key；

3. 复制 API 端点地址，格式一般为：
   `https://你的Dify域名/v1/chat-messages`

### 2. 新建自定义 HTTP 工具

回到「告警预处理 Agent」的工具管理，新建自定义 HTTP 工具：

* **工具标识**：`call_main_ops_agent`

* **工具名称**：调用主排障智能体

* **工具描述**：当识别到 P1、P2 级故障组时，调用此工具将故障组发送给主排障智能体，进行深度根因排查与修复方案生成。

* **请求方式**：POST

* **请求 URL**：你复制的主排障 Agent API 端点地址

* **Headers**：新增 2 个请求头
  
  * `Authorization`：`Bearer 你的主Agent API密钥`
  * `Content-Type`：`application/json`

* **请求体（Body）**：JSON 格式，参数化传入故障信息
  json
  
      {
        "query": "{{fault_info}}",
        "response_mode": "streaming",
        "user": "alert-system"
      }

* **参数配置**：新增 1 个必填参数
  
  * 参数名：`fault_info`
  * 类型：字符串
  * 描述：聚合后的故障组完整信息，包含根因告警、关联告警、影响范围

### 3. 工具启用与规则强化

1. 在预处理 Agent 的工具列表中，勾选启用这个工具；
2. 在系统提示词中补充一条规则：**P1、P2 级故障组必须调用「调用主排障智能体」工具，传入完整故障组信息；P3 级告警禁止调用该工具**。

第三步：监控系统 Webhook 接入预处理 Agent
----------------------------

让监控系统产生的告警，自动推送到预处理 Agent，实现全链路自动化。

### 1. 获取预处理 Agent 的 Webhook 地址

1. 进入「告警预处理 Agent」→【发布】→【API 访问】，创建 API 密钥；
2. 复制对话 API 端点，作为告警推送的目标地址。

### 2. 监控系统配置（以 Prometheus Alertmanager 为例）

在 Alertmanager 配置文件中，新增 webhook 接收器，指向预处理 Agent 的 API：

yaml
{}    receivers:
      - name: 'agent-webhook'
        webhook_configs:
          - url: 'https://你的Dify域名/v1/chat-messages'
            http_config:
              authorization:
                type: Bearer
                credentials: '你的预处理Agent API密钥'
            send_resolved: true}

其他监控系统（Zabbix、Grafana）同理，在告警通知里新增 Webhook 通道，地址填 Dify 的 API 端点。

### 3. 简化方案（MVP 二期推荐）

如果不想直接改监控配置，先用**中间转发脚本**做一层适配：

* 写一个简单的 Python/Shell 脚本，接收监控告警，整理成文本格式，再调用预处理 Agent 的 API；
* 好处是可以做格式兼容、失败重试，避免监控告警格式不兼容导致 Agent 解析失败。

第四步：全链路自动流转规则（成本控制关键）
---------------------

为避免无效调用浪费 token，建议按等级设置不同处理策略：

1. **P1 级故障**：预处理完成后，**自动调用**主排障 Agent 深度排查，同时推送告警通知给运维；
2. **P2 级故障**：预处理完成后，**自动调用**主排障 Agent，结果推送给运维参考；
3. **P3 级告警**：仅聚合输出清单，**不自动调用**主排障 Agent，每日汇总一次推送即可。

三、3 组验收测试用例
===========

直接输入原始告警文本，验证预处理能力是否达标。
用例 1：重复告警压制验证

-------------

* **输入**：

plaintext
    【P2】192.168.1.20 CPU使用率96%
    【P2】192.168.1.20 CPU使用率97%
    【P2】192.168.1.20 CPU使用率95%

* **预期输出**：合并为 1 条持续告警，故障组数量为 1，无重复条目；原始告警总数为 3，已过滤 / 合并说明清晰。

用例 2：连锁告警聚合验证
-------------

* **输入**：
    【P2】192.168.1.20 CPU使用率96%
    【P2】订单接口响应超时率18%
    【P1】MySQL主库慢查询260条/分钟
    【P2】支付接口超时率12%
    【P3】日志服务磁盘使用率85%

* **预期输出**：
  
  1. 生成 2 个故障组：
     
     * G001：P1 级，根因为慢查询，关联 CPU 高、订单 / 支付超时，自动触发深度排查；
     * G002：P3 级，磁盘告警，不自动排查；
  
  2. 故障主题、影响范围描述准确，因果关系正确。

用例 3：无效告警过滤验证
-------------

* **输入**：
    【P2】测试环境-用户服务接口超时
    【P1】生产环境-主库CPU使用率98%
    【P2】测试环境-日志服务异常

* **预期输出**：测试环境告警被标记过滤 / 降级，核心生产故障置顶为 P1，优先级重排序正确。

四、优化技巧与避坑指南
===========

1. **聚合准确率提升**：如果业务链路复杂，可给预处理 Agent 增加一个「CMDB 服务依赖查询」工具，让它能查询服务上下游关系，聚合因果时更精准，不会误判。
2. **成本控制**：预处理 Agent 只做语义分析，不调用重工具，token 消耗极低；核心成本在主排障 Agent，通过分级触发策略，可减少 70% 以上的无效调用。
3. **格式不稳定排查**：如果模型偶尔输出非 JSON 格式，一是开启 Dify 结构化输出强制校验，二是在提示词末尾增加「输出必须是纯 JSON，不能有 markdown 代码块标记」的约束。
4. **影子模式过渡**：上线初期建议先开启「只聚合不自动调用」，预处理结果只推送给运维人工验证，准确率稳定后再打开自动调用主 Agent 的开关。
